# Phase 4 設計書 — 広告 ROAS（費用対効果）

- 日付: 2026-07-14
- 対象: Sup! Sup! マーケティング分析ダッシュボード / Phase 4
- ステータス: **設計のみ（着手条件待ち）**。広告を実際に出稿していないため実装は保留。

---

## 0. 先に読むこと — 構造的な制約と、その回避策

**厳密な広告ROAS（広告→予約の1対1帰属）は、この事業構成では原理的に測れません。**

- 予約完了は**アソビュー（`supsup.urkt.in`）＝外部ドメイン**で発生する（Phase 2 で確認済み）。
- 広告プラットフォーム（Google/Meta）のコンバージョン計測は、自社サイトにピクセルを置いて成果を測る仕組み。**アソビューの予約完了ページにピクセルは置けない**。
- したがって「このキャンペーンが何件の予約を生んだ／ROAS◯倍」という厳密な数字は出せない。**出せると謳う設計は誤り**。

→ 本Phaseは**3層の"近似帰属"**を重ねて、実務判断に足る精度を作る（§3）。

---

## 1. 目的

広告出稿を始めた際に「その広告費は回収できているか／どの広告が効いているか」を判断できるようにする。

---

## 2. 【重要】広告を待たずに今できること — 自己申告の流入経路

**既に取得済みなのに使っていないデータがあります。**

`sync/src/parser.ts` は予約CSVから以下を取得しているが、`history:latest` には含めていない:
- **`howFound`（`●ご予約の経緯`）** … お客様の自己申告による「どこで知ったか」
- **`media`（`媒体`）** … 予約媒体

これは**予約実績に直結した帰属データ**であり、広告の有無に関係なく「流入経路・認知の拡大」（当初4つの問いの1つ）に直接答えます。GA4のセッションやSNSのリーチが「認知の入口」の推測であるのに対し、**`howFound` は実際に予約した人の申告**＝最も予約に近い。

### 提案：Phase 1.2（広告より先にやる価値が高い・小さい追加）
- **sync**: `HistoryRecord` に `howFound` と `media` を追加（`buildHistoryRecords` に2行）。※自由記述なのでPII混入に注意 → **正規化＋既知カテゴリへの丸め**を行い、生の自由記述はKVに入れない方針を推奨（例: 「Instagram」「検索」「知人紹介」「リピート」「その他」）。
- **dashboard**: 「自己申告の流入経路」カード（経路別の予約数・売上）＋ 経路×時系列。
- 効果: **広告を出す前から、どの経路が実際の予約を生んでいるかが分かる**。広告開始後は「広告経由」の申告も拾える。

> Phase 4 に着手する前に、この Phase 1.2 を先に入れることを**強く推奨**。

---

## 3. Phase 4 の帰属モデル（3層）

厳密帰属が不可能な前提で、次の3つを重ねて判断する。

| 層 | 何を見るか | 精度 | 実現方法 |
|---|---|---|---|
| **A. 時系列の重ね描き** | 広告費・クリックと、予約実績を同一時間軸で | 粗い（相関） | Ads API の費用/クリック × `history:latest` |
| **B. UTM → GA4 キャンペーン別送客** | 広告クリック→`supsup.jp`（UTM付き）→ GA4でキャンペーン別セッション | 中（送客量まで） | Phase 2 の GA4 資産を拡張（`sessionCampaignName` 次元） |
| **C. 自己申告（`howFound`）** | 予約者本人が「広告で知った」と申告 | **予約に最も近い** | Phase 1.2（§2） |

- **粗ROAS** = 期間の予約売上 ÷ 期間の広告費（全体）。**キャンペーン別の厳密ROASは出さない**（出せないため）。
- 代わりに「キャンペーン別の送客量（B）」と「自己申告での広告寄与（C）」を並べ、費用配分の判断材料にする。

## 4. アーキテクチャ

Phase 2/3 と同型（dashboard が API を直接叩く＋DASH KVキャッシュ＋未接続フォールバック）。

### 4.1 データ源と認証（難易度が大きく違う）

| 媒体 | API | 認証の重さ | 備考 |
|---|---|---|---|
| **Meta広告** | Marketing API（`/insights`: spend, impressions, clicks, cpc をキャンペーン別・日別） | **軽い**：**Phase 3 で作る Facebook アプリを流用**し、権限に `ads_read` を追加するだけ | → **Meta から着手するのが合理的** |
| **Google広告** | Google Ads API | **重い**：developer token の申請・審査、OAuth（サービスアカウント不可の場合が多い） | Meta の後で必要になれば |

→ **Phase 4 は Meta 広告から**。Google広告は必要性が出てから別途。

### 4.2 コンポーネント（新規）
- `src/ads/meta.ts` — Marketing API `/insights` 取得（Phase 3 の `igGet` と同型のラッパを流用/共通化）。日別・キャンペーン別の spend/impressions/clicks。
- `src/metrics/roas.ts` — 広告費 × 予約売上の月次重ね描き＋粗ROAS（純）。
- `src/ads/section.ts` — 広告カード群（未接続フォールバック付き）。
- GA4側の拡張（層B）: `CAMPAIGN_SPEC`（`sessionCampaignName` × sessions）を `ga4/reports.ts` に追加。

## 5. 画面（Phase 4）

1. **広告費 × 予約売上の重ね描き**（月次。棒=広告費 / 線=予約売上）＋**粗ROAS**（売上÷広告費）
2. **キャンペーン別の費用・クリック・CPC**（Meta）
3. **キャンペーン別の送客**（GA4 `sessionCampaignName` 別セッション＝層B）
4. **自己申告の流入経路に占める「広告」**（層C・Phase 1.2 の資産）
5. **広告インサイト**（決定論。「粗ROAS ◯倍」「CPC が前月比 +X%」「広告経由の申告が予約の Y%」等）＋**厳密帰属ができない旨の注記をUIに明示**

## 6. スコープ外
- キャンペーン別の厳密ROAS／アトリビューションモデリング（構造上不可能）。
- 広告の入稿・自動最適化（分析専用）。
- Google広告（当面は Meta のみ）。

## 7. 実装の進め方
- Phase 1/2/3 と同様：メイン＝設計・監査・レビュー、実装はサブエージェント委譲、TDD＋モック、未接続フォールバック。
- 難所＝**帰属の限界をUIで誠実に表現すること**（誤読を招く「ROAS」表示を避ける）。

## 8. 着手条件（これが揃うまで実装しない）

1. **実際に広告を出稿していること**（Meta広告アカウントに費用・クリックの実績があること）。出稿前に作っても表示するデータが無い。
2. Phase 3 の Facebook アプリが存在すること（`ads_read` 権限を足して流用）。
3. 広告のリンク先に **UTMパラメータ**を付けて `supsup.jp` に送客していること（層B に必要）。
4. （推奨）**Phase 1.2（自己申告の流入経路）が先に入っていること**（層C の土台）。
